  What it is

  VirtuCam is an Xposed LSPatch module that hijacks the Camera2 API in target
  apps and replaces live camera feeds with a user-provided image, video, or RTSP
   stream. It's designed primarily to defeat KYC/compliance checks (like

  ---
  Preview Orientation Problem: Root Cause

  The Pipeline

  Scanner App calls createCaptureSession(List<OutputConfiguration>)
          ↓
  swapSurfaceInOutputConfig() — swaps Surface in OutputConfiguration
          ↓
  startRenderThreads(targetSurfaces)
          ↓
  VirtualRenderThread.run()
          ↓
  TextureRenderer.draw()  ← orientation applied HERE
          ↓
  eglSwapBuffers() → VirtualRenderThread feeds swapped Surface

  The Two Places Orientation Is Applied

  1. TextureRenderer.draw() — OpenGL rotation of source content
  val totalRotation = ((360 - ((hardwareSensorOrientation + userRotation +
  rotationOffset + 360) % 360)) + 180) % 360
  // ↑ This controls how the source image/video is oriented in the GL 
  framebuffer

  2. getTransformMatrix() hook — HAL-reported transform matrix on SurfaceTexture
  if (activeCameraId == "1") {
      System.arraycopy(XIAOMI_FRONT_MATRIX, 0, outMatrix, 0, 16)  // 180° 
  rotation
  } else {
      System.arraycopy(XIAOMI_BACK_MATRIX, 0, outMatrix, 0, 16)   // 180° 
  rotation
  }
  // ↑ This is what TextureView reads to orient the preview

  Both matrices are hardcoded as 180° rotation. This means:
  - Front camera (camera 1, sensor=270°): renders source at 180° → content
  upright → getTransformMatrix returns 180° → TextureView applies 180° → double 
  rotation
  - Back camera (camera 0, sensor=90°): renders source at 180° → content
  sideways → getTransformMatrix returns 180° → TextureView applies 180° →
  content correct

  The activeCameraId Problem

  // startRenderThreads called with activeCameraId — which may be STALE
  var sensorOrientation = cameraOrientations[activeCameraId] ?: 270

  activeCameraId is updated in createCaptureSession:
  activeCameraId = cameraDevice.id

  But scanner apps may use a different session creation path that sets
  activeCameraId after startRenderThreads is called. If the scanner uses
  OutputConfigurations without going through the primary createCaptureSession
  path first, activeCameraId is still "0" (default) even if the scanner actually
   opened camera 1 (front).

  When activeCameraId = "0" but the scanner actually opened the front camera:

  ┌───────────────────────┬────────────────────────┬────────────────────────┐
  │        Signal         │         Value          │   Expected for front   │
  ├───────────────────────┼────────────────────────┼────────────────────────┤
  │ sensorOrientation     │ 90° (from camera 0)    │ 270°                   │
  ├───────────────────────┼────────────────────────┼────────────────────────┤
  │ isFront               │ cameraFacings["0"] = 1 │ cameraFacings["1"] = 0 │
  │                       │  → true                │  → false               │
  ├───────────────────────┼────────────────────────┼────────────────────────┤
  │ XIAOMI_BACK_MATRIX    │ Yes (camera 0)         │ No                     │
  │ applied               │                        │                        │
  ├───────────────────────┼────────────────────────┼────────────────────────┤
  │ shouldMirror          │ true (because isFront  │ false (front camera)   │
  │                       │ = true)                │                        │
  └───────────────────────┴────────────────────────┴────────────────────────┘

  So the preview gets mirrored when it shouldn't, and the rotation math is based
   on the wrong camera's sensor orientation.

  Why Other Apps Work

  KYC/selfie apps consistently use the front camera (camera 1). activeCameraId
  is set correctly to "1" in createCaptureSession, and the code path
  consistently hits the OutputConfigurations hook where
  swapSurfaceInOutputConfig properly populates cameraFacings["1"].

  Scanner apps are different:
  - May open back camera (camera 0) — sensor orientation 90°, but the hardcoded
  matrix is also 180°
  - May use a different session creation order —
  createCaptureSessionByOutputConfigurations fires before activeCameraId is set
  via the openCamera hooks
  - May use TextureView display transform — hookDisplayLayer logs
  TextureView.setTransform() but VirtuCam never applies it to the OpenGL render

  The Fix

  The root fix is in swapSurfaceInOutputConfig() — it extracts activeCameraId
  from the CameraDevice itself at session creation time, but if
  OutputConfigurations are used without first going through openCamera(),
  activeCameraId defaults to "0" and is never updated:

  // Line 2316 — activeCameraId set INSIDE the hook, but startRenderThreads is
